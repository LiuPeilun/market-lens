## 项目目标

Market Lens 是面向个人投资研究的智能体项目。用户既可以通过代码或名称直接查询股票、基金和 ETF，也可以通过自然语言对话触发数据查询、确定性分析和工具调用。

项目输出用于研究辅助，不构成投资建议。LLM 负责理解问题和解释结果，估值计算、数据质量判断和风险控制必须由可测试的确定性代码完成。

## 核心原则

1. **先验证数据，再设计因子**：只有经过实际接口请求、字段语义检查和覆盖率验证的数据，才能进入正式评分。
2. **数据路由必须准确**：基金、ETF、ETF 联接基金和指数必须先解析真实跟踪关系；无法确认时应停止估值，不允许用错误持仓兜底。
3. **不同概念独立输出**：估值、基本面质量、产品质量和置信度不能混为一个未经解释的分数。
4. **缺失数据不猜测**：缺失因子应降低覆盖率和置信度，不由 LLM 补全，也不静默使用行业均值替代。
5. **行业采用专属模型**：银行、保险、券商、周期、成长和红利类资产不能共用完全相同的因子权重。
6. **所有结论可追溯**：输出应包含数据日期、来源、计算方法、因子值、覆盖率和降级原因。
7. **高风险工具必须受控**：MCP 和代码执行统一经过 ToolRegistry、风险策略、审批、审计与沙箱边界。

## 当前估值实现

### 股票

旧版兼容字段 `valuation.score` 仍使用以下历史分位权重：

| 因子 | 当前权重 |
| --- | ---: |
| PE TTM | 40% |
| PB | 30% |
| PS TTM | 15% |
| PCF OCF TTM | 15% |

V2 顶层 `assessment` 当前使用 `model_version=valuation-v2.2.0-fund-product-models`，股票仍按财务接口的 `ORG_TYPE` 优先路由，行业分类只作回退，不根据股票名称判断。当前模型如下：

| 模型 | 估值因子 | 基本面质量因子 |
| --- | --- | --- |
| 通用非金融 | 历史 PE/PB/PS/PCF，同板块 PE/PB 分位 | ROE、ROIC、净利润增长、营收增长、ROE 稳定性 |
| 银行 | PB 为主，历史 PE，同板块 PE/PB 分位 | ROE、净息差、不良率、拨备覆盖率、资本充足率、ROE 稳定性 |
| 保险 | PB 为主并辅以历史 PE；暂不使用行业分位 | ROE、偿付能力、新业务价值增长及利润率、净利润增长、ROE 稳定性 |
| 券商 | PB 为主，历史 PE，同板块 PE/PB 分位 | ROE、风险/流动性覆盖、净稳定资金率、净资本/净资产、净利润增长、ROE 稳定性 |

估值分方向为 `0` 极度低估、`100` 极度高估；质量分方向为 `0` 较差、`100` 较好。两者独立计算，质量变化不会改写估值等级。股票产品质量固定为 `null`，综合吸引力在回测前固定为 `null`。

### 基金与 ETF

- 普通基金目前使用持仓加权 PE/PB 历史分位、同行 PE 分位和股息率。
- 红利低波基金提高股息率权重。
- 指数/ETF 在无法获得指数估值历史时使用指数价格历史分位代理，该结果只能表示价格位置，不能等同于真实基本面估值。
- 当前稳定持仓路径通常只能获得前十大持仓。覆盖率不足时必须降低置信度，不能宣称代表完整基金组合。
- 产品路由区分 ETF、ETF 联接、普通指数基金和主动基金；目标 ETF 与跟踪指数关系优先于名称回退。
- ETF、ETF 联接和普通指数基金的产品质量使用年度费率 30%、期末规模 20%、年化跟踪误差 30%、绝对年化跟踪偏离 20%。跟踪误差是核心因子，至少需要 `60` 个共同收益日，`252` 日达到完整样本。
- 主动基金产品质量使用年度费率 60% 和期末规模 40%，不强行计算不适用的跟踪指标。
- 基金底层资产质量使用前十大持仓股票的 V2 质量分加权计算；覆盖率和路由类型只影响置信度，不把前十大包装成完整组合。
- 产品费率只包含管理费、托管费和销售服务费；期末规模采用对数锚点，第一版锚点与权重等待 V2-7 回测校准。

## 已确认的输出模型

最终输出拆分为四个独立维度：

| 维度 | 判断问题 | 分数方向 |
| --- | --- | --- |
| 估值分 | 当前价格相对自身历史和同行业贵不贵 | `0` 极低估，`100` 极高估 |
| 基本面质量分 | 盈利能力、经营稳定性和成长可持续性如何 | `0` 较差，`100` 较好 |
| 产品质量分 | 基金/ETF 的费率、规模、流动性和跟踪质量如何 | `0` 较差，`100` 较好；股票不适用 |
| 置信度 | 数据和模型对此结果的支持程度有多高 | `0` 无可靠结论，`100` 高可靠性 |

成长因子纳入基本面质量分，不再额外制造一个含义重叠的第五维度。产品质量不能改变底层资产估值，置信度也不能作为投资吸引力奖励项。

### 估值等级

| 估值分 | 等级 |
| ---: | --- |
| 0 至小于 15 | 极度低估 |
| 15 至小于 30 | 低估 |
| 30 至小于 45 | 正常估值偏下 |
| 45 至小于 60 | 正常估值 |
| 60 至小于 75 | 正常估值偏上 |
| 75 至小于 90 | 高估 |
| 90 至 100 | 极度高估 |

## 待回测假设：综合吸引力分

原“调整分”候选名称为“综合吸引力分”。它不是估值分，也不能覆盖四个独立维度，只能作为多标的研究排序假设。回测通过前不进入正式 API、前端或 LLM 结论。

以下公式仅作为待验证假设，不是生产权重：

```text
便宜程度 = 100 - 估值分

股票原始吸引力 = 便宜程度 * 60% + 基本面质量分 * 40%

基金/ETF 原始吸引力 =
    便宜程度 * 50%
  + 底层资产质量分 * 30%
  + 产品质量分 * 20%

置信度调整后吸引力 =
    50 + (原始吸引力 - 50) * 置信度
```

当置信度为 `0` 时，候选结果回归中性 `50`；置信度为 `1` 时，保留原始值。正式权重、等级和展示文案必须由 V2-7 回测确定，V2-8 才允许发布。任何综合分都必须与四个原始维度同时展示，不能映射为买入或卖出信号。

## 估值优化实施工作流

估值 V2 按“数据事实先于评分逻辑”的顺序实施。任何工作包都不得跨过数据验证直接修改生产权重。

### 单个因子的准入流程

1. **实时探测**：使用多个行业代表性标的请求真实接口，确认字段存在、非空覆盖、日期和单位。
2. **保存夹具**：将脱敏后的典型响应、空值响应和异常响应固化为测试夹具，不让单元测试依赖公网。
3. **建立类型**：在 `market_lens/types.py` 中定义明确字段，不让评分逻辑直接读取 `raw` 字典。
4. **解析与诊断**：数据适配器输出字段值，同时记录来源、日期、状态和降级原因。
5. **只读展示**：新字段先进入分析上下文和 API，经过代表性标的核对后才允许加入评分。
6. **评分接入**：定义适用模型、方向、归一化、异常值、缺失值、权重和最小样本。
7. **回归验证**：运行解析、指标、API、LLM 上下文和前端类型测试，并更新本文档。

### 目标数据流

```text
Eastmoney / CSI / Fund data source
  -> typed data adapter
  -> source status and freshness diagnostics
  -> normalized factor observations
  -> asset and industry profile router
  -> valuation / quality / product scorers
  -> confidence engine
  -> optional attractiveness projection after backtesting
  -> analyzer result
  -> API / Supabase history / LLM context / frontend
```

LLM 只接收最终结构化结果和诊断，不参与模型路由、数据补全、因子归一化或分数计算。

### 目标结果契约

估值 V2 新增顶层 `assessment`，使估值、基本面质量、产品质量和置信度保持语义独立。迁移期间保留现有 `valuation.score`、`level`、`level_zh`、`confidence` 和 `factors` 作为纯估值兼容字段。

```json
{
  "assessment": {
    "schema_version": "2",
    "model_version": "valuation-v2",
    "profile": "generic_non_financial",
    "analysis_as_of": "2026-07-21",
    "dimensions": {
      "valuation": {"score": 42.0, "confidence": 0.85, "factors": []},
      "quality": {"score": 71.0, "confidence": 0.50, "factors": []},
      "product": null
    },
    "overall_confidence": 0.50,
    "attractiveness": null,
    "confidence_detail": {
      "components": {},
      "caps": [],
      "reasons": []
    },
    "data_quality": {
      "sources": [],
      "warnings": [],
      "source_as_of": null,
      "retrieved_at": null
    }
  },
  "valuation": {
    "score": 42.0,
    "level": "slightly_undervalued",
    "confidence": 0.85
  }
}
```

股票的 `product` 固定为 `null`，不能用交易活跃度伪造产品质量。基金和 ETF 的 `quality` 表示底层资产质量，`product` 表示基金载体质量。各维度分别计算置信度，`overall_confidence` 使用保守聚合，不能用高置信度维度掩盖低置信度维度。

### 因子结果契约

每个正式因子至少输出以下信息：

| 字段 | 含义 |
| --- | --- |
| `key`、`name`、`category` | 唯一标识、展示名称和所属维度 |
| `value`、`unit`、`source_as_of` | 原始值、单位和数据日期 |
| `score`、`direction` | 标准化分数和高低方向 |
| `normalization` | 历史分位、行业分位或行业专属锚点 |
| `weight`、`effective_weight` | 模型权重和缺失处理后的实际权重 |
| `sample_size`、`coverage` | 样本量和数据覆盖率 |
| `source`、`status`、`eligible` | 来源、可得状态和是否满足计分条件 |
| `warnings` | 负值、异常值、样本不足或口径限制 |

只有 `status=available`、`eligible=true` 且满足最小样本的因子可以计分。`not_applicable` 不扣覆盖率，`missing/stale/error` 必须影响覆盖率或置信度。数据存在但样本不足时保持 `status=available`、设置 `eligible=false`，避免混淆“拿不到数据”和“暂不适合评分”。

### 评分基础规则

以下规则先作为实现约束，阈值在回测阶段校准：

- PE、PCF 等只在正值时参与“便宜程度”分位；亏损不能因为负 PE 被判为低估。
- PB、PS 也要求有限正值；异常值先按可解释规则过滤，不能由单次极值主导分数。
- 历史估值分位至少需要约 `252` 个交易日，约 `1200` 个交易日达到完整历史样本置信度。
- 同板块横截面使用东方财富估值板块代码；有效样本少于 `10` 时只展示原始值，不形成分位。`ORIG_BOARD_CODE` 是同板块旧代码别名，不是一级行业代码，不能用于父级回退。
- 财务质量优先使用至少 `3` 个年度报告观察稳定性；不足时可展示最新值，但降低置信度。
- 分红稳定性至少使用 `3` 个完整年度；仅有近 12 个月股息率时不声称“持续分红”。
- ETF 跟踪误差至少需要 `60` 个重叠交易日，达到 `252` 日后才视为完整年度样本。
- 缺失因子允许在同一维度内重新归一化权重，但必须满足该模型的核心因子门槛和最低有效权重。
- PEG 暂不进入正式核心评分，避免负增长、预测口径和极值造成错误排序。

### 置信度规则

置信度必须独立于资产好坏，由以下组件共同决定：

1. 数据源请求是否成功。
2. 数据日期是否满足对应数据类型的新鲜度要求。
3. 核心因子和总因子覆盖率。
4. 历史、行业和财务样本量是否充分。
5. 基金跟踪关系与持仓路由是否完整。
6. 数据是否存在冲突、异常值或代理方法。

每个维度单独计算置信度，总置信度采用保守聚合并允许设置上限：指数价格位置代理不能获得与真实指数估值相同的置信度；仅前十大持仓的主动基金不能获得完整组合估值的高置信度；跟踪关系未解析时估值置信度应为 `0`。

### 时间与版本语义

- `analysis_as_of`：本次分析允许使用数据的截止日期。
- `source_as_of`：行情、财报、持仓或基金资料本身的数据日期。
- `retrieved_at`：系统实际获取该数据的时间。
- `schema_version`：API 结果结构版本。
- `model_version`：评分配置、归一化和阈值版本。

新鲜度必须相对 `analysis_as_of` 判断，不能固定相对当前日期，否则历史分析和回测会错误使用未来信息或把当时有效的数据判为过期。

### 模型路由顺序

模型路由必须是确定性代码，并输出选择理由：

1. 先区分股票、主动基金、指数基金、ETF 和 ETF 联接基金。
2. 股票按行业优先路由到银行、保险、券商或通用非金融模型。
3. 通用非金融模型再附加红利、成长、周期等风格标签，但标签不能覆盖行业模型。
4. 基金先解析跟踪指数和目标 ETF，再决定底层资产估值与产品质量路径。
5. 无法可靠分类时使用通用模型并降低置信度，不由名称关键词强行选择高风险专属模型。

- EV/EBITDA、研发效率、保险 PEV 等当前不可稳定获得的指标不进入第一版核心模型。
- 指数价格分位必须标记为价格位置代理，不能称为真实指数估值。

### 2026-07-21：V2-3 评分基础设施

- V2 结果使用 `schema_version=2` 和 `model_version=valuation-v2.0.0-infrastructure`，相同版本和输入必须产生相同输出。
- 因子状态统一为 `available/missing/stale/error/invalid/not_applicable`；数据存在但样本不足时保持 `available` 且 `eligible=false`。

- PE 和 PB 是当前通用股票估值维度的核心因子；核心因子缺失或有效权重低于 50% 时，新维度不计分。
- 缺失因子在维度内重归一化有效权重，原始权重和实际权重同时输出；`not_applicable` 不降低覆盖率。
- 维度置信度由来源、时效、因子覆盖、数据覆盖、样本充分度、路由和方法质量独立计算，并允许代理方法设置上限。

- 迁移期间保留旧估值字段；基金旧因子带 `legacy_compatibility_factor` 标记，不伪装成已完成的新产品模型。

### 2026-07-21：V2-5 基金与 ETF 产品模型

- API 使用显式 V2 `assessment` 响应模型公开三类维度、总体置信度、来源和诊断，同时允许现有业务扩展字段继续透传。
- 历史分析契约将 `assessment` 设为可选；旧记录回退到 `valuation` 的估值分、等级和置信度，新记录展示独立维度。
- 前端主界面不再把估值称为“综合估值”，而是分别展示估值位置、基本面/底层资产质量、基金产品质量和总体置信度。
- LLM 上下文保留完整 `assessment` 的因子、日期、来源和诊断；固定模板与系统提示禁止把独立维度合并成未经回测的吸引力或买卖结论。
- `attractiveness` 继续保持 `null`，前端不显示占位评级，后续是否发布由 V2-7 回测结果决定。

### 2026-07-21：V2-7A 回测基础设施

- 新增独立回测输入与报告契约；混合模型版本、重复快照、未来来源日期、非有限分数及未经时点验证的数据会直接失败。
- 入场固定使用 `analysis_as_of` 后首个可用收盘价，禁止使用分析日同日价格；持有期按交易会话计数。
- 所有评分先统一成“越高越有吸引力”方向：估值使用 `100 - 估值分`，质量和产品质量保持原方向。
- 报告包含分层收益、信息系数、分层单调性、最大回撤、换手、评分稳定性以及行业、模型、市场阶段和置信度分组。
- 综合吸引力候选只在训练日期中选型，再在后续验证日期中评估；即使达到门槛也只标记为可进入 V2-8 人工审核。
- 当前 API 生成的普通分析结果不自动获得回测资格；输入必须由历史采集器明确写入时点验证来源。
- 基金报告日不等于披露日。在历史披露时间和历史产品快照可验证前，不运行基金模型参数校准。

### 2026-07-21：V2-7B 股票历史截面采集器

- 新增 `stock-universe-1` 清单契约；必须声明时点验证、包含退市成员并提供无重叠的历史成员区间和历史行业区间。
- 采集器按月末或季末寻找 7 天内最近交易日，分别截断价格、估值、行业横截面和财务数据。
- 财务指标必须具有 `notice_date` 且公告日不晚于分析日；缺少公告日的历史财务行不会进入模型。
- 样本使用历史清单提供的行业分类，不读取当前公司行业后倒推历史；行业估值仍使用对应交易日估值记录中的板块代码。
- 默认严格模式下，任一成员快照因价格、估值、成员或行业缺失被跳过都会使采集失败；`--allow-partial` 只用于数据源排查。
- 新增 `collect-stock-backtest` CLI 和 `docs/BACKTESTING.md`，记录清单格式、采集/报告命令与已知限制。
- 仅完成采集能力不等于已有真实回测证据；可信历史样本清单和退市终止价值处理仍是下一项工作。

## 文档维护规则

每次后续工作都应遵循以下流程：

1. 开始任务前阅读本文档，确认当前阶段、既有决策和下一项工作。
2. 开发过程中如发现数据源、字段语义或方案假设有误，先修正文档中的事实。
3. 完成任务后更新当前能力、V2 工作包状态、验证结果和决策日志。
4. 规划项和已完成项必须分开书写，不得把“接口存在”描述为“功能已实现”。
5. 提交代码时，文档更新应与对应功能处于同一提交或紧随其后的文档提交。
